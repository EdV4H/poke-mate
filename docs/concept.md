# コンセプト: 共有ワークスペース

## 目的

**Pokémon Champions**（2026年4月8日発売、Switch/Switch2、モバイル版は2026年夏）の対戦を**より深く考える**ための思考支援ツール。GUIの視覚的操作性とAIの分析力を、同じデータモデルの上で融合させる。

基本プレイ無料・バトル特化のタイトルで、過去作からPokémon HOME経由で「出張」してきたポケモンと、作中でスカウトしたポケモンで戦う。従来作（特にSV）とは対戦仕様が大きく異なるため、**Champions固有の仕様に最適化する**のが本アプリの方針。

## 3つの芯

4機能を別々のアプリに見せず、**1つの「対戦研究ノート」** として統合する。

- **Workspace**: ユーザーの作業空間。複数の Party/BattleSession を束ねる最上位コンテナ。
- **Artifact**: Workspace に属する「モノ」。Party / BattleSession / BattleLog / MetaSnapshot / Note すべて Artifact の派生。
- **View**: 各 Artifact を操作する GUI 画面（構築/対戦/ログ/メタ）。

> AIが `update_party_slot` を呼ぶと、開いているPartyビューが即光って更新される。GUIとAIが「同じオブジェクトを編集している」感覚を作るのがこのアプリの根幹。

## Pokémon Champions の対戦仕様（本アプリの前提）

- **Lv50固定**（育成なし、レベル表示なし）
- **個体値は全員31固定**（変更不可）
- **努力値は廃止、代わりに「ステータスポイント（SP）」システム**。HOME転送時に `4 EV → 1 SP`（上乗せで `8 EV → +1 SP`）で換算される
- **技・特性・性格・持ち物はトレーニング（VP消費）で自由変更**可能
- **メガシンカのみ実装**。キーアイテムは**ゼンブイリング**。Zワザ/ダイマックス/テラスタルは**2026年4月時点では非対応**（将来追加予定）
- **ランク/カジュアル/プライベート** の3モード、各モードで**シングル/ダブル**を選択
- **シングルは6匹持ち3匹選出、ダブルは6匹持ち4匹選出**、試合時間最大20分
- **対戦プールは約278種類**（初期）。既存作からHOME経由で連れてくるか、作中スカウトで入手
- **技・特性・道具はChampions独自バランス**（例: 新特性 Dragonize / Mega Sol / Piercing Drill / Spicy Spray、技の威力/命中/分類変更多数、状態異常の弱体化）
- **ランク制**: モンスターボール級→スーパーボール級→ハイパーボール級→マスターボール級→チャンピオン級。月次シーズン制
- **HOMEから「出張」**: Championsで行った変更はHOME側に戻らない（GOからは片道）

## MVPでカバーする4機能

1. **パーティ構築支援**: 6匹のパーティ編成、技/持ち物/特性/性格/SP配分の設計。AIが相性・役割・SP最適化・選出パターンを分析。
2. **対戦シミュレーション**: Champions仕様のダメージ計算（Lv50固定、SP由来実数値、独自バランス反映）、選出読み、メガシンカタイミング検討。
3. **対戦ログ振り返り**: リプレイ/ログを読み込み、AIが分岐点（選出ミス、メガタイミング、交代択）を指摘。
4. **環境メタ分析**: シーズンごとの使用率、メガ別採用率、流行の技構成をAIが可視化・解説。

## 典型的なユーザー動線

```
[メタ画面] 今シーズンの流行メガとトップ構築を確認
    ↓ 「ボックスのメガリザードンYを軸に組みたい」
[新規Party] GUIで6匹ポチポチ（HOMEから出張した手持ちから選択）
    ↓
Claude Desktopで「このパーティの穴と3匹選出パターンを分析して」
    ↓
review-party Skill が起動
  → MCP: get_party → analyze_type_coverage → get_meta_snapshot
  → analyze_selection_patterns（3匹選出のマッチアップ表）
    ↓
AI: 「対トップ5構築で勝率高い選出は A+B+C。鋼偏重で●●が重いので
     6枠目を▲▲に変える提案」
  → update_party_slot / update_training で提案を反映
    ↓
GUIが即座に更新（トースト「AIが変更しました」）
    ↓
[シミュレーション] メガシンカタイミングと選出読みを検証
    ↓
[試合後] リプレイをD&D
  → analyze-battle-log が分岐点（選出/メガタイミング/交代択）を指摘
```

## 設計原則

- **Single Source of Truth**: SQLiteが真実。GUIもMCPもそれを見る。
- **Single writer**: 書き込みは必ず `DataService` 経由。GUI直SQL禁止、MCPもDataServiceを呼ぶ。
- **Change events**: `change_events` テーブルをメッセージキューとして使い、GUI↔MCP の状態同期を取る。
- **Optimistic UI + Reconcile**: GUIはローカル楽観更新、DBコミット後に差分照合。
- **AIの動きが見える**: AIによる書き込みはGUIに明示的にフィードバック（トースト、ハイライト）。
