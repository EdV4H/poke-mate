# 主要リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| **Champions独自バランスの追跡負荷** | ダメージ計算・AI提案が実機と乖離 | `champions-diff.json` を第一級データとし、ゲーム更新ごとにバージョン管理。技/特性/道具の変更点（例: Beak Blast強化、Snap Trap鋼化、状態異常弱体化、新特性4種）を網羅するテストを用意 |
| **リプレイ/バトルログ形式が未公開** | ログ機能が実装できない | Phase 0では**手動入力/テキスト貼り付け/画面ログ**から開始。公式リプレイ形式が明らかになり次第 `log-parser` にアダプタ追加。形式非依存な中間表現を設計 |
| **対戦プール（初期278種）が今後拡張される** | マスタデータが陳腐化 | `champions_available` フラグをマスタに持ち、データパック更新で追加。UI側でもシーズンバッジ表示 |
| **Zワザ/ダイマックス/テラスタル将来解禁** | 設計変更 | 現時点では持たず、解禁時に `tera_type` 列などをマイグレーションで追加。mechanicsは抽象化（`mega` `z_move` `dynamax` `tera` を`transformation_kind`で扱う）して列挙を拡張するだけで済むようにする |
| **HOME出張の片道/双方向の混在仕様** | 同じポケモンの重複管理 | `pokemon_sets.origin` / `origin_meta_json` でHOME/スカウトを区別。GO出身は片道フラグ、その他は同期可能フラグを立ててUIに示す |
| **SP換算の仕様精緻化** | 既存構築から移行した値が実機とズレる | HOME転送ロジック（4EV→1SP, +8EV→+1SP）をテスト網羅。ゲーム側の端数処理が判明次第差分反映 |
| **ダメージ計算の正確性** | ユーザー信頼性が崩壊 | `packages/damage-calc` を独立させ、`@smogon/calc` をフォーク→Champions仕様に改造。既知ケース（ダメージ表/実機ログ）で 50 件以上のユニットテスト。Lv50・IV31固定なので式は簡潔 |
| **MCP と GUI の状態同期複雑化** | バグの温床 | Single writer 原則（DataService 経由のみ）、`change_events` テーブルで監査、楽観ロック。Renderer は「サーバ状態の投影」と割り切る |
| **Claude Desktop 更新で MCP 仕様が変わる** | 動かなくなる | MCP SDK をピン留め、`@modelcontextprotocol/inspector` で E2E テストを CI 化 |
| **Skill が長すぎて動作不安定** | UX 劣化 | 1 Skill は 10 ステップ以内。複雑な処理は MCP ツール側に寄せる（計算はツール、判断は Skill） |
| **SQLite 同時書き込み競合（Electron + standalone MCP）** | データ破損 | WAL モード、書き込みは常に DataService 経由、`BEGIN IMMEDIATE` + `busy_timeout` + 指数バックオフのリトライ。standalone MCP は Claude Desktop が起動する別プロセスのため Electron から自動 shutdown は前提にしない。pid入り `lockfile` で他方の起動を検知し、競合時はユーザーへ警告 / 切替案内を出す |
| **Electron 配布のコード署名・公証** | インストール不可 | 最初から `electron-builder` + Apple 公証フロー。Phase 0 で一度配布まで通す |
| **Zustand / IPC スキーマ不整合** | 型エラー地獄 | `@poke-mate/shared-types` パッケージで IPC チャンネル契約を一元化 |
| **モバイル版クロスプラットフォーム化（2026年夏）時のアプリ棲み分け** | アプリのスコープ曖昧化 | 本アプリはあくまで**思考支援/分析ツール**（対戦相手はゲーム本体）。モバイル版発売後もポジションは変わらない |

## 早期検証が必要な仮説

Phase 0 のうちに潰しておきたい技術検証:

1. **MCP サーバーが Electron 子プロセスと standalone の両モードで動くか**
2. **SQLite WAL で複数プロセス同時アクセスが壊れないか**
3. **macOS 公証が通る Electron ビルドを作れるか**
4. **Claude Desktop の `claude_desktop_config.json` に自動マージして認識されるか**
5. **Championsのリプレイ/バトルログを何らかの形で取り込めるか**（画面キャプチャOCR? 手動入力? URLシェア?）— 実機での確認必須

## データパックの持続性

Championsのマスタデータは「誰がメンテするか」が長期リスク。公式データダンプがないため、ゲーム側アップデートの追従が恒常的な課題。

- **短期**: 手動で維持（Bulbapedia / GameWith / ポケモン徹底攻略などを参照）
- **中期**: コミュニティデータパックに依存（Pokémon Showdown系プロジェクトのChampions対応フォーク）
- **長期**: マスタエディタをアプリ同梱し、ユーザー貢献で更新できるように。差分PRをGitHubで受けられる仕組みを検討

## ゲームの進化に対する追従計画

Championsは今後 **Zワザ → ダイマックス/キョダイマックス → テラスタル** の順で対応メカニクスが拡張される可能性が高い（公式も示唆）。各機能追加時にダメージ計算・マスタデータ・UIの3領域で影響が出るため、スキーマと計算エンジンに**抽象層**を用意しておき、新メカニクス追加はデータパック差し替え+新ルール実装だけで済ませる。
